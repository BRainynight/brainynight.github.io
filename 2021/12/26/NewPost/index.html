Module and Package

這篇的重點依然在於「相對引用與絕對引用」，但在這之間，先來看一些有關 import 的觀念。

[TOC]

Concept

  • Python 只有一種型態的 module object，所有的 Module 都是這種型態(type)。不管它是用 C, Python 或其他來實作。

  • 管理這些 module ，提供他們名稱層級架構的，就是 Package。

  • 可以把 Package 就想像成一個檔案系統管理檔案的樣子，但 Package 的管理並不如 file system 這樣的單純。

    所有的 package 本質上都是 module，但不是所有的 module 都是 package ，這是一個充分非必要條件。Package 可以視為是一種特殊的 modeuld，更仔細的說：任何包含有 __path__ 屬性的 module 會被認為是一個 package。

    All packages are modules, but not all module are packages.

    Packages Are a special kind of module; specifically, any module that contains a __path__ attribute is considered a package.

  • Python定義了兩種的 package，regular package 和 namespace package

Regular package

現在的 python 不一定非得有 __init__.py 才能是 package，但是只要有這個檔案，在 import 到該 package 時，一定會先執行 __init__.py 再去執行其他動作。

如下面的例子中，並沒有直接 import parent，都只有匯入其下的 subpackage，然而，parent 下面有一個 __init__.py ，所以必定會先執行。接著才是 subpackage 的 __init__.py。

也因此，如果有一些初始化動作要做，可以放在 __init__.py，它就如同其他一般的 Python Module 一樣。

  • 傳統的 package，只要資料夾中含有 __init__.py ，當 regular package 被 import 時，__init__.py 會被偷偷地先執行，並且，這個檔案裏面的物件，會被綁定在 package 的名稱空間下。
  • 當 __init__.py 被 import 時，Python 會添加一些額外的 attribute 進來。

Namespace packages (簡介)

  • namespace package 是由多個 portions (直接翻譯：部份)構成的，每個 portion 都貢獻一個 subpackage 給 parent package。
  • portion 可分佈在 file system 中不同的地方，資料夾、網路、甚至是 zip file 反正必須是 Python 可以搜尋到的地方。
  • Namespace packages 的 __path__ 不使用傳統的 list，而是一個自訂的可迭代物件類型(custom iterable type)
    • 若 parent 的路徑發生改變，這個可迭代物件在下一次的 import 中，會自動重新搜尋 package portions 。
  • namespace packages 沒有 parent/__init__.py 這樣的檔案，事實上，在 import search 的時候，他們可能會找到多個 parent directories 。不同的 parent directories 中都提供了不同的 subpackage。
  • 以上面的例子來說， parent/one 與 parent/two 在檔案架構中的物理位置，可以不是相鄰的兩個資料夾。
  • 關於 Namespace Packages 可以看 PEP 420 有更多細節。

Import Search

import 有兩個步驟，這裡只著重在 Search。為了開始搜索，首先需要導入 module/package 的完整限定名稱，這個名稱會在導入搜索的各階段被使用。

1.Module Cache

  • 在 import search 期間，首先會被檢查的地方是 sys.modules。
  • import search 的範圍就是 sys.modules。這個字典會將已經載入的模組名稱(module name) mapping 到已經加載的 module 實例。
  • sys.modules 是可寫的，但是要很小心、隨意改寫很容易有錯，總之不要動它。
    • 隨意地刪除 key 有可能不會破壞關聯的 Module (因為其他 module 可能保留對它的reference)。但是，他會使字典的映射功能無效﹔使用該 key 找 module 時找不到，引發 ModuleNotFoundError

sys.modules 是一個字典，紀錄著 modeul name : module instance 這樣一組一組的訊息，是匯入模組時第一個搜尋的地方。

2. Finders and Loader

若指定的 module name 在 sys.modules 中找不到，會調用 Python 的 import 協議 (protocol)，以尋找並加載此 module 。

這個協議包含兩個部分: finders and loaders，但將不細講。

PEP328 Imports: Multi-Line and Absolute/Relative

Absolute Import 絕對位置匯入

absolute import 指被引入的 package 使用的是從 root folder 到該 package/module 之間的完整路徑。

像這樣的用法，在 Python 2.4 之前，並沒有明確的說 import 所指的位置應該是 top-level 或是 其他 module insider。為了解決這個歧異(ambiguity) 從 Python 2.5 開始，這種 import foo 的寫法，被稱為「 absolute import」。

它代表這個 foo 必定是可以透過 sys.path 取得的 module 或 package，這稱為絕對匯入。

To resolve the ambiguity, it is proposed that foo will always be a module or package reachable from sys.path. This is called an absolute import.

根據 PEP 328，從 python 2.5開始，所有直接import xxx 語法，都會是 top-level module : 與 __main__ 同級(資料夾)的 module。

絕對導入的優點是清楚、架構易懂，但也有著問題：重命名層次結構中 high-level 的 package 時、或是將一個 package 移至另一個 package 時，要去修改所有的程式碼，需要付出較高的代價。

強制把該 package 加到被搜索行列（ sys.path）裡面。

Relative Import

當 import package 時 package 前面有 . 就是相對匯入。

在最標準的起手式import .pkgName ，相對路徑的起點就是寫這行的檔案的所在位置而非 root file 。

相對匯入的原則 (Rationale for Relative Imports)

隨著絕對匯入的作法敲定，出現了一個問題﹔是否該允許相對匯入的存在。相對匯入最大的優點在於：重新編排大型 package 架構時，不需要一一編輯 sub-package。

此外，沒有相對引入的話，在 package 內部的模組，無法輕易的 import 自己。(In addition, a module inside a package can’t easily import itself without relative imports.) 看看最上面 movieTheater 的案例，明明相鄰的兩個檔案，卻需要大老遠從 root 拉路徑過來，這種繞遠路的行為十足弔詭。

缺點

然而，使用相對路徑還是有些須注意的地方：

  1. 使用了相對路徑的檔案沒辦法直接執行(不能作為 __main__)
  2. 由於相對路徑變得要計算套件之間的上下層關係來決定 import，因此 top level 不能低於被 import 的套件。需要在 import 套件中最高層的位置的再上一層，建立呼叫這一整套 package 的 .py 作為直接執行的檔案（他就是 __main__）。

某些程度上這種作為變得十足麻煩，想測試套件還需要往上爬一層創立另一個檔案再來 import …. ，甚至有些專案會把相對路徑跟絕對路徑都寫出來：

我曾經在 stackoverflow 看到一則討論在說為何用了相對路徑的 python 檔不能執行？這很麻煩。而某位網友說：「使用相對路徑的檔案你應該視為『套件』而非『腳本 (script) 』」。

或許正因為這個原因，比起嫌相對路徑比絕對路徑麻煩，更應該把相對路徑跟絕對路徑本來當作是在不同的場合所使用的…. 這樣的思路來思考吧。

例如，如果 visualize\prining.py 這個檔案的相對路徑只用到了同層級的 ( .py also in visualize )，那相對路徑在搜索 visualize 名稱空間可以找到這個包，那直接使用 python testfile.py 應該就沒什麼問題。

但是， visualize\prining.py 的相對路徑 call 的是上一層資料夾中的檔案，我們不能夠失去這個「上層資料夾」的名稱，否則無法找尋目標檔案，那就必須再往上一層去呼叫這個 package 裡面的包，以保留給他們完整的名稱空間。

Why Relative Import?

前面說了一些 import 的流程。使用絕對路徑 import 起來很清晰，但有些缺點，像是如果很多層，一層層這樣 import 顯得很冗長。

但最大的問題在於 — import package 不可以重複的名稱，否則，運氣好是找不到 module 報錯、運氣不好是誤用其他同名的 module 而不自知。

Attribute : __package__¶

以下是官方文檔的內容：

The module’s __package__ attribute must be set. Its value must be a string, but it can be the same value as its __name__. When the module is a package, its __package__ value should be set to its __name__. When the module is not a package, __package__ should be set to the empty string for top-level modules, or for submodules, to the parent package’s name. See PEP 366 for further details.

This attribute is used instead of __name__ to calculate explicit relative imports for main modules, as defined in PEP 366. It is expected to have the same value as __spec__.parent.

模組的 __package__ 在相對匯入時必須設置一串 string，他可以與 __name__ 同名。當 module 是一個 package 的時候，他的 __package__ 應該與 __name__ 相同。若模組不是 package，若他是 top-level modeules 則 __package__ 應該為空字串、若是子模組則應該是其 parent 的名字。更多細節參見 PEP 366。

這個屬性將會代替__name__來計算 module 在相對匯入時的實際位置。
